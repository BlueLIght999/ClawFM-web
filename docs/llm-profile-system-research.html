一、核心结论

结论：直接用 LLM 构建"数据采集 → 画像生成 → 分析决策"体系，三大功能（个性推荐、用户聚类、Agent 画像感知）均可通过 DeepSeek API + 现有架构实现，无需引入新基础设施。

功能技术路径关键参考实现复杂度
个性推荐LLM 从播放历史+聊天记录生成结构化画像 JSON，补齐 topGenres/analysis 空缺，增强 recommender 排序PURE 框架[1] + RecGPT 淘宝猜你喜欢[4]中等
用户聚类LLM 零样本分类生成语义标签（"深夜情绪听众"），可选用 LLM embedding + K-means 做粗粒度分桶ClusterLLM[12] + Clustering-with-LLM[13]低
Agent 画像感知混合模式：核心画像全量注入 system prompt（Profile-as-Context）+ 新增画像工具按需检索（Profile-as-Tool）LangGraph LongTermMemoryAgent[17] + Self-RAG 自适应检索[22]中等

3.3 权重融合策略

参考 Spotify 多模型融合[5]和 Last.fm chosenByUser 区分[6]：

行为类型权重信号强度
主动搜索并完整收听×3最强偏好信号
DJ 推荐未跳过×1表示可接受
跳过行为×-1明确不喜欢
聊天显式表达（"我想听 xx"）×5（需衰减）极强但有时效性
收藏/加歌单×4强偏好信号

3.4 离线批处理 + 缓存

遵循 RecGPT 的核心原则[4]：LLM 做重的离线理解，传统算法做快的在线检索。

  • 画像生成：离线运行（每小时或每 20 首歌触发一次），结果缓存到 user_profile 表
  • 推荐检索：实时用传统算法（recommender 的 rankSongsByTopArtists），读取缓存的画像
  • 推荐解释：需要时实时调用 LLM 生成（可缓存常用解释）

四、功能2：用户聚类划分

4.1 范式选择

ClusterLLM 证明了 LLM 可以通过自然语言指令指定聚类视角[12]。对于本项目，采用范式 C（LLM 直接零样本分类）最务实——因为项目是单用户电台，聚类目标是为 DJ 话术和推荐策略提供上下文标签，而非大规模用户分桶。

范式方法适用场景本项目适用性
A: LLM 指南三重任务指导嵌入器聚类3K-50K 规模文本聚类过重
B: LLM 嵌入 + K-meansLLM 生成 embedding → K-means混合特征用户分群未来可选
C: LLM 直接分类自然语言定义类别 → LLM 归类小规模、需可解释标签首选

4.2 分群类别定义（自然语言）

LLM 能理解语义化标签[12][14]，而非 K-means 的无意义编号：

类别定义（Few-shot Prompt 注入）：

- 深夜情绪听众：常在 22:00-02:00 收听，偏好民谣/独立/慢歌，
  完整听完率高，skip 率低
- 流行音乐爱好者：偏好热门榜单，流派以流行为主，白天收听为主，
  主动搜索比例高
- 通勤节奏听众：工作日 07:00-09:00/17:00-19:00 集中收听，
  偏好节奏感强的曲风，session 短
- 探索型乐迷：流派分散，主动搜索比例高，skip 率适中，
  常试听新歌手
- 怀旧金曲派：偏好 80-00 年代歌曲，重复收听率高，
  新歌接受度低

4.3 动态分群机制

参考 PURE 的增量画像更新[1]和 2026 年 Agent Memory 综述的画像漂移处理[22]：

  • 定期重分类：每次画像刷新时重新评估 listenerCategory
  • 漂移检测：比对新旧 category，变化时记录到 chat_history 供 DJ 参考
  • 衰减因子：旧行为权重随时间衰减（参考 Ebbinghaus 遗忘曲线[22]）
  • 置信度阈值：低置信度（<0.5）标记为"过渡期"，DJ 话术更保守

4.4 聚类结果消费

消费方使用方式
DJ 话术生成根据 listenerCategory 调整语气和选曲策略（深夜情绪听众 → 更温柔的话术）
推荐策略不同类别用不同的填充策略权重（探索型 → 更多新歌，怀旧派 → 更多经典）
主动播报根据类别决定播报频率和内容（通勤听众 → 少打断，探索型 → 多介绍新歌）
Agent 工具决策用户说"放点好听的"时，根据类别推断具体偏好

5.2 Profile-as-Context 增强

现有 context.js slotMemory 仅注入 topArtists(5)。扩展为完整核心画像：

// context.js slotMemory 扩展
function slotMemory(repositories) {
  const profile = repositories.profile.get();

  // 核心画像（全量注入，<500 tokens）
  const coreProfile = [
    profile.topArtists?.slice(0, 5).map(a => `- ${a.name}`).join('\n'),
    profile.listenerCategory
      ? `### 听众类型\n${profile.listenerCategory} (置信度: ${profile.categoryConfidence})`
      : '',
    profile.analysis
      ? `### AI 画像分析\n${profile.analysis}`
      : '',
    profile.moodPreferences
      ? `### 时段偏好\n${JSON.stringify(profile.moodPreferences)}`
      : '',
    profile.skipPatterns
      ? `### 跳过模式\n${profile.skipPatterns}`
      : '',
  ].filter(Boolean).join('\n\n');

  return coreProfile;
}

5.3 Profile-as-Tool 新增工具

新增 2 个 agent 工具，注册到 ToolRegistry[17]：

工具名描述参数返回
get_profile_detail获取用户详细画像信息（按需检索）{ section: "history" | "mood_rules" | "skip_patterns" | "all" }对应画像 section 的详细文本
update_preference用户明确表达偏好时更新画像{ type: "like" | "dislike", artist?: string, genre?: string, note?: string }{ updated: true }

5.4 画像感知的意图消歧

当前 mergedIntentPromptBuilder.js 仅注入 topArtists[15]。扩展为多维度画像，实现多粒度意图解析[15]：

用户输入无画像时有画像时
"放点好听的"search_music("热门")recommend() — 基于深夜情绪听众类别推断偏好轻柔民谣
"换一首"skip()skip() + 记录跳过模式到画像
"今天心情不好"search_music("伤感")recommend({preference:"舒缓"}) — 基于 moodPreferences.night
"这个歌手不行"skip()skip() + update_preference({type:"dislike", artist}) + 调整推荐权重

5.5 画像漂移处理

参考 2026 Agent Memory 综述[22]的 Write-Manage-Read 闭环：

阶段机制实现方式
Write事件驱动写入跳过/收藏/聊天 → 实时更新画像权重
Manage定期摘要+衰减+矛盾消解每小时 LLM 重新生成画像，旧偏好应用衰减因子
ReadCore 全量 + Archival 按需system prompt 注入核心画像 + 工具检索详细画像

画像漂移风险：Agent Memory 综述指出"Manage 是工程上最难的环节"[22]。Self-RAG 的自适应检索模式可减少不必要的 token 消耗——Agent 自己决定何时需要检索详细画像[22]。反思式更新需 reflection grounding（每条反思必须引用具体证据），避免自我强化错误[22]。

## 输出要求
请以 JSON 格式输出用户画像，包含以下字段：

8.2 延迟分析

8.3 风险与缓解

风险严重度缓解方案
LLM 幻觉：推断不存在的行为中prompt 约束"仅基于提供的数据分析" + JSON Schema 验证
画像漂移：偏好变化未被及时捕获中事件驱动更新（跳过/收藏实时触发）+ 每小时定期刷新
画像过时：旧偏好权重过高低衰减因子（Ebbinghaus 遗忘曲线）+ 矛盾消解
Token 膨胀：画像注入增加 prompt 长度低Core 画像控制在 500 tokens 以内，详细画像走 Archival 工具
